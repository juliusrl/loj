# Loj 環境変数 / プロジェクト体系 (Project Story)

## ステータス

このドキュメントでは、第 2 Beta ゲートに向けて実装済みの現在のベースラインについて説明します。

実装済みの現在のベースライン：

- `loj validate`、`loj build`、および `loj dev` はすべて、`loj.project.yaml` の近くにある標準的なプロジェクト環境変数ファイルをロードします。
- 共有環境変数ファイルがサポートされています：
  - `.env`
  - `.env.local`
- ターゲットスコープの環境変数ファイルがサポートされています：
  - `.env.<target-alias>`
  - `.env.<target-alias>.local`
- `loj dev` はこれらの環境変数ファイルを監視し、変更時にターゲットセッションおよび管理下の Host/Server プロセスをリロードします。
- プロジェクト全体の Host/Server の規約は、限定的な `LOJ_DEV_*` 環境変数レイヤーを通じてオーバーライドできるようになりました。

意図的に保留されている機能：

- `loj.project.yaml` 内部での任意の環境変数テンプレート化。
- 1 つのプロジェクトファイル内での複数の Host/Server スタック。
- 非標準的なターゲットごとのプロセススクリプト記述。

---

## 目的

環境変数の体系は、`loj.project.yaml` を汎用的なデプロイ設定ファイルに変えることなく、ローカルのフルスタック開発を予測可能にする必要があります。

以下の質問に答えるものであるべきです：

- ローカル設定はどこに置くべきか。
- どの値をチェックインしても安全か。
- どの値をローカル専用にするか。
- 共有環境変数とターゲットスコープ環境変数が、生成されたターゲットにどのように到達するか。

オーケストレーションをシェルスクリプト DSL に変えてはなりません。

---

## ファイルの発見

`loj` は `loj.project.yaml` と同じディレクトリから環境変数ファイルをロードします。

共有ファイル：

- `.env`
- `.env.local`

ターゲットスコープファイル：

- `.env.<target-alias>`
- `.env.<target-alias>.local`

例：

```text
examples/fullstack-user-admin/
  loj.project.yaml
  .env
  .env.local
  .env.frontend
  .env.frontend.local
  .env.backend
  .env.backend.local
```

---

## 優先順位

現在の優先順位は以下の通りです：

1. `.env`
2. `.env.local`
3. `.env.<target-alias>`
4. `.env.<target-alias>.local`
5. 呼び出し元プロセスからのシェル環境変数

解釈：

- チェックインされる共有デフォルト値は `.env` に記述します。
- マシンローカルの共有オーバーライド値は `.env.local` に記述します。
- チェックインされるターゲットデフォルト値は `.env.<target-alias>` に記述します。
- マシンローカルのターゲットオーバーライド値は `.env.<target-alias>.local` に記述します。
- エクスポートされたシェル環境変数は、ファイルベースの環境変数よりも優先されます。

## 限定的なプロジェクトキー

プロジェクトシェルは現在、少数の `LOJ_DEV_*` キーを認識します：

- `LOJ_DEV_HOST`
- `LOJ_DEV_HOST_PORT`
- `LOJ_DEV_HOST_PREVIEW_PORT`
- `LOJ_DEV_API_BASE`
- `LOJ_DEV_PROXY_AUTH`
- `LOJ_DEV_SERVER_HOST`
- `LOJ_DEV_SERVER_PORT`

これらのキーは、`dev.host` / `dev.server` または現在のデフォルトから取得される標準的なローカル体験の値をオーバーライドします。

これらは意図的に限定されています：

- `loj.project.yaml` への任意のキー展開(interpolation)はありません。
- コマンドのテンプレート化はありません。
- プロジェクトファイル内に汎用的な環境変数バッグ(env bag)はありません。

## ターゲットスコープの注入

各ターゲットは以下を受け取ります：

- `.env` / `.env.local` からの共有環境変数
- `.env.<target>` / `.env.<target>.local` からのターゲットスコープ環境変数

現在の動作：

- `loj validate` および `loj build` は、有効なターゲット環境変数を委譲されたターゲット CLI に渡します。
- `loj dev` は、有効なターゲット環境変数を委譲されたターゲット開発ループに渡します。
- 管理下のローカルプロセスもターゲット環境変数を受け取ります：
  - フロントエンドホストはフロントエンドターゲット環境変数を受け取ります。
  - バックエンドサーバーはバックエンドターゲット環境変数を受け取ります。

`loj dev` はまた、いくつかのコンパイラ所有のローカル変数を管理プロセスに注入します。例：

- `RDSL_GENERATED_DIR`
- `VITE_RDSL_API_BASE`
- `RDSL_PROXY_API_TARGET`
- `RDSL_PROXY_API_AUTH`
- `SERVER_ADDRESS`
- `SERVER_PORT`
- `LOJ_PROJECT_FILE`
- `LOJ_PROJECT_DIR`
- `LOJ_TARGET_ALIAS`
- `LOJ_TARGET_TYPE`

同じキーの場合、コンパイラ所有の変数がファイルベースの環境変数よりも優先されます。

## プロジェクトファイルの境界

これらは `loj.project.yaml` に保持してください：

- ターゲットのメンバーシップ
- ターゲットのエントリファイル
- 出力ディレクトリ
- ホスト/サーバーのトポロジ
- 安定した開発シェル構造

これらは環境変数ファイルに保持してください：

- ローカルプロキシの資格情報
- ローカルのホスト/サーバーポートのオーバーライド
- ターゲットローカルなフレームワーク環境変数
- マシン固有の統合設定

これらは `loj.project.yaml` に**入れないでください**：

- 回避可能なパスワードやトークン
- ターゲットごとの任意の環境変数バッグ
- シェルコマンド
- デプロイ固有のインフラ状態

## チェックイン vs ローカル

推奨ルール：

- 値が機密情報ではなく、サンプルをすぐに実行するのに役立つ場合は、`.env` をチェックインしても構いません。
- `.env.local` および `.env.<target>.local` はローカル専用に留めるべきです。

現在のリリースサンプルのパターン：

- `examples/fullstack-user-admin/.env` はチェックインされています。これは、デモ用の Basic 認証資格情報がローカルの MVP の利便性を目的としたものであり、真の秘密ではないためです。
- 実際の機密情報は、引き続きローカル専用の環境変数ファイルに保持する必要があります。
